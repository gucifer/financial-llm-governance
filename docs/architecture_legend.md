# Architecture Flow Legend

Each numbered arrow in `financial_llm_governance_architecture.png` corresponds to a step below.

| # | From | To | What happens | FS AI RMF Pillar |
|---|------|----|--------------|------------------|
| ① | AML/KYC & Regulatory Q&A clients | Workload Router & Risk Scorer | Incoming requests are tagged by workload type (AML vs. Regulatory Q&A) and assigned a data-sensitivity risk tier (low / medium / high). | Pillar 2 — Risk Identification |
| ② | Workload Router & Risk Scorer | Prompt Injection Detector | Risk-tagged request is scanned for adversarial inputs — prompt injection, jailbreak patterns, and OWASP LLM Top 10 #1 attack vectors. Malicious requests are rejected before reaching the gateway. | Pillar 2 — Risk Identification |
| ③ | Prompt Injection Detector | Kong Gateway · JWT/OAuth 2.0 · PII Scrubbing | Sanitized request fans into the on-prem DMZ layer. Kong enforces JWT/OAuth 2.0 authentication; the PII middleware strips SSNs, account numbers, and ABA routing numbers before any data leaves the perimeter. | Pillar 1 — Governance · Pillar 3 — Controls |
| ④ | Kong layer | Semantic Cache (Redis + Embeddings) | Scrubbed, authenticated request is checked against a vector-based semantic cache. If a semantically equivalent query was answered recently, the cached response is returned — reducing Azure OpenAI token spend and latency. | Pillar 3 — Controls |
| ⑤ | Semantic Cache | AWS API Gateway · AWS WAF · Rate Limiter | On cache miss, the request crosses the cloud boundary. AWS WAF applies rule-based filtering; the rate limiter enforces per-client quotas and writes a structured request log for Pillar 4 audit. | Pillar 2 — Risk Identification · Pillar 4 — Monitoring |
| ⑥ | Rate Limiter | pgvector (RAG — Regulatory Q&A path) | Regulatory Q&A requests are routed to the vector database, which stores chunked embeddings of the FS AI RMF 230 control objectives, FINRA rulebook, SEC guidance, and FinCEN advisories. | Pillar 1 — Governance |
| ⑦ | Rate Limiter | Hybrid AML Model (AML/KYC path) | AML/KYC transaction narratives are routed to the hybrid supervised + unsupervised anomaly detection pipeline for risk scoring prior to LLM processing. | Pillar 2 — Risk Identification |
| ⑧ | pgvector | Cross-Encoder Re-ranker | Retrieved regulatory document chunks are re-ranked by a cross-encoder model to surface the most relevant passages. Only the top-ranked context is injected into the LLM prompt — reducing hallucination risk from irrelevant context. | Pillar 3 — Controls |
| ⑨ | Hybrid AML Model | False-Positive Reduction Engine | Initial AML risk scores pass through the false-positive reduction engine (target: <10% false-positive rate vs. the industry baseline of 90–95%). Transactions below threshold are cleared; high-risk transactions carry a structured risk annotation into the LLM prompt. | Pillar 2 — Risk Identification |
| ⑩ | Cross-Encoder Re-ranker · FP Reduction Engine | Azure OpenAI (GPT-4o) | Both paths converge. The grounded regulatory context (Reg Q&A) or risk-annotated transaction narrative (AML) is sent to the sovereign Azure OpenAI endpoint for inference. No raw PII reaches this layer. | Pillar 3 — Controls |
| ⑪ | Azure OpenAI | ECLIPSE Hallucination Detector · PII Re-check · NeMo Guardrails | LLM response is fanned to all three output validators in parallel: (1) ECLIPSE detects factually incorrect regulatory citations; (2) PII re-check ensures the model did not regenerate scrubbed data from context; (3) NeMo Guardrails enforces STR field schema compliance. | Pillar 3 — Controls |
| ⑫ | Output Validation layer | Langfuse · SHAP/LIME · CloudWatch · SIEM | Validated responses are forwarded to the observability pipeline. Langfuse captures full LLM traces; SHAP/LIME generates per-decision feature attributions formatted as FINRA-compliant audit JSON; CloudWatch records metrics; SIEM ingests all events for security alerting. | Pillar 4 — Incident Response · Pillar 1 — Governance |
| ⑬ | Observability layer | MLflow Registry · Human Review Queue · Automated Eval Trigger | Audit records and model performance signals are written to the governance layer. MLflow stores model versions and model cards; anomalous signals enqueue human review; the eval trigger schedules re-evaluation runs against the hallucination detection suite. | Pillar 1 — Governance |
| ⑭ | Automated Eval Re-run Trigger | ECLIPSE Hallucination Detector *(dashed)* | Feedback loop: when human reviewers flag an incorrect output or the eval trigger fires, a re-evaluation job reruns the ECLIPSE hallucination suite against the flagged response. Results update the model card in MLflow and inform retraining decisions. | Pillar 4 — Incident Response |
